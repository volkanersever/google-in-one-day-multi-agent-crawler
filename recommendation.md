# Production Deployment — Recommendations

The shipped system runs on one box using plain files and the Node.js standard library, which is deliberate for the exercise and for grader-hand-inspection of the data format. Moving this into a real production search product would proceed in three tiers. **Tier 1 — keep the architecture, harden the substrate.** Put each component behind a supervisor (`systemd`, `pm2`, or a container with health checks), front the HTTP API with an nginx reverse proxy that terminates TLS and adds per-IP rate limiting, and move the on-disk files onto a tier-1 SSD with daily snapshotting. Add structured logging (pino / JSON lines) and ship `/status` + crawler events to Prometheus via a tiny exporter; dashboards should cover pages/sec, fetch-error rate by host, back-pressure time-in-state, and letter-file growth slope. At the same time, implement `robots.txt` fetch+cache per host, respect `Crawl-delay`, and enforce a configurable crawl budget per origin. Politeness and a real User-Agent with a contact URL are table stakes before crawling anything beyond our own corpus.

**Tier 2 — split storage and scale search separately from crawling.** Letter files are perfect for inspection but wrong for scale: `s.data` already grows faster than `q.data` by two orders of magnitude, and there is no compaction. Migrate the word index to a proper inverted-index store — either a trie-backed LMDB/RocksDB key-value (postings per token as the value) or, if the team is happy with it, an OpenSearch cluster; the search module already abstracts the read path through `iterateLetter`, so swapping the backend is a one-file change. The visited-URL set should move to a Redis Set for atomic cross-worker dedup, and crawl-state JSON files should become small records in the same KV. Once storage is externalized, the crawler becomes a stateless worker: run a fleet of Node processes across regions, each pulling from a shared frontier queue (SQS / Kafka / Redis Streams — any of them fits), each applying its own token-bucket rate limit per host, and all writing through the shared inverted-index service. Search becomes its own fleet behind a load balancer, reading (read-only) from the same index and caching hot queries at the edge. This separation lets search availability and latency SLOs be met independently of crawler load — which matters, because search is user-facing and a crawler stall should never take it down. On the compliance side, URL normalization rules must include PII-stripping for query strings before indexing, `/index` should gate on an authenticated admin role, and all storage must be at-rest encrypted with a documented retention window. Tier 3 is where you consider ranking beyond exact-match (PageRank-style link graph signals, BM25 over TF-IDF, query understanding) and that conversation belongs with the product team, not the infrastructure runbook.
